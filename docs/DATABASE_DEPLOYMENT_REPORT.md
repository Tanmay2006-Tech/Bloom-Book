# DATABASE DEPLOYMENT REPORT

## 1. Exact database architecture

BloomBook uses PostgreSQL (Neon in production), Drizzle ORM at runtime, and Drizzle Kit for schema generation and deployment. The canonical TypeScript schema is `lib/db/src/schema/index.ts`, which exports all eight table modules in the same directory. The root `drizzle.config.ts` points exclusively to that canonical schema and writes versioned SQL plus metadata to `drizzle/`.

All tables live in PostgreSQL's `public` schema. They are independent journal collections, so there are currently no foreign keys or cross-table relations. Each table has a text UUID primary key generated by the application, required content fields where appropriate, and a timezone-aware `created_at` timestamp defaulting to `now()`.

The production API currently mirrors these physical definitions inside `vercel-deploy/api/index.ts` to keep the Vercel function self-contained. Migration changes must start in `lib/db/src/schema`; regenerate the SQL afterward and keep the deployed API model aligned.

Drizzle records applied production migrations in `public.__drizzle_migrations`. This bookkeeping table is created automatically by `drizzle-kit migrate` and is not an application-domain table.

## 3. Exact Neon command

Put the Neon pooled connection string in `.env.local`:

```dotenv
DATABASE_URL=postgresql://USER:PASSWORD@HOST/DB_NAME?sslmode=require
```

Then apply the checked-in migration with exactly:

```bash
npm run db:migrate
```

For a brand-new empty Neon database, this also creates the current schema directly:

```bash
npm run db:push
```

No package-directory change and no extra `--config` flag are required. The root config loads `DATABASE_URL` from the current process first, followed by `.env.local` and `.env`.

## 4. Migrations versus push

Use **`npm run db:migrate` for production**. The SQL is versioned, reviewable, repeatable, and tracked in `__drizzle_migrations`. This is the recommended command for Neon and CI/CD release workflows.

Use **`npm run db:push` for disposable development databases or a deliberate one-time empty Neon bootstrap**. Push compares live state with TypeScript schemas and applies changes directly, but does not provide the same audited release history. Never run `push --force` against production data.

Do not run database migration automatically inside every Vercel frontend build. Concurrent Preview/Production builds can race, and builds should not mutate production infrastructure. Run the migration once as an explicit release step, then deploy or redeploy the API.

## 5. Configuration status

- Root `drizzle.config.ts`: configured for PostgreSQL, all schemas, `drizzle/` output, verbose/strict mode, and public migration metadata.
- Environment loading: supports shell `DATABASE_URL`, `.env.local`, and `.env`; missing credentials produce an immediate readable error for push/migrate/studio.
- Initial migration: generated at `drizzle/0000_mute_karen_page.sql` with eight tables and twelve indexes.
- npm scripts: `db:generate`, `db:migrate`, `db:push`, `db:studio`, and `db:verify` are available at the repository root.
- Legacy nested config: removed so there is one migration source of truth.
- Local migration execution: verified successfully with `npm run db:verify` against an embedded PostgreSQL-compatible engine.
- Live Neon execution: not performed in this workspace because no `DATABASE_URL`, `.env`, `.env.local`, PostgreSQL CLI, Docker runtime, or Neon connector was available. Once the connection string is present, `npm run db:migrate` is the only required deployment command.

## Operational workflow

When changing a schema:

```bash
npm run db:generate
npm run db:verify
npm run db:migrate
```

After the first Neon migration, verify the deployed application API:

```bash
npm run smoke:live-api
```
